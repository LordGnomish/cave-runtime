# Cave Runtime Dev Mode — Implementation Roadmap

**Date:** 2026-05-28
**Owner:** Burak Tartan
**Governing ADR:** [ADR-RUNTIME-DEV-MODE-001](../adr/ADR-RUNTIME-DEV-MODE-001.md)
**AI ayağı:** [ADR-RUNTIME-OPENJARVIS-ADOPTION-001](../adr/ADR-RUNTIME-OPENJARVIS-ADOPTION-001.md)

Bu doküman `cave-runtime --mode=dev`'in faz faz implementasyon planıdır. Her faz bağımsız bir ray olarak dispatch edilebilir; fazlar arası bağımlılık aşağıda işaretli.

---

## İlkeler (red lines)

- **Tek binary.** `--mode` aynı `cave-runtime` artifact'ı üzerinde bir runtime flag'i. Yeni binary yok, microservice topology yok, helm-per-crate yok.
- **Parity-first.** Dev mode ve server mode ortak reconcile/codepath'leri paylaşır; dev-only fork yasak.
- **Honest degradation.** macOS/Windows fallback datapath "not production-equivalent" diye net işaretlenir.
- **Strict TDD.** Cave golden rule: RED→GREEN, port effort üstlenilir.

---

## Phase 1 — Profile system

**Hedef:** `--mode=dev` flag'i + profil preset iskeleti + component opt-in.

- `--mode={server|dev}` argümanı (default `server`, mevcut davranış korunur).
- Profil preset'leri: `minimal` (~4 GB), `standard` (~8 GB), `full` (~16 GB+).
- Component opt-in registry: hangi Layer 3/4 component'i hangi profilde default-on/off.
- Minimum çekirdek: `apiserver + kubelet + cri` her profilde açık.
- Graceful degradation iskeleti: opt-out edilen component'in "yokken makul davran" yolu.
- **OpenJarvis hedef crate kararı** burada netleşir (cave-hermes alt-modül vs cave-agent) — ADR-RUNTIME-OPENJARVIS-ADOPTION-001 "hedef crate notu" bu fazda kapanır.

**Bağımlılık:** yok (temel). **Çıktı:** çalışan `cave-runtime --mode=dev --profile=minimal`.

## Phase 2 — `cavectl dev` suite

**Hedef:** developer CLI UX.

- `cavectl dev up [--profile=...]` — dev runtime başlat.
- `cavectl dev down` — durdur.
- `cavectl dev status` — component sağlığı + kaynak kullanımı (+ fallback uyarısı).
- `cavectl dev logs [component]` — birleşik / filtreli log stream.
- `cavectl dev exec <workload> -- <cmd>` — workload içinde komut.
- `cavectl dev shell <workload>` — interaktif shell.
- `cavectl dev reset` — `~/.cave/` state temizle.

**Bağımlılık:** Phase 1 (profil + component lifecycle). **Çıktı:** tam dev inner-loop CLI.

## Phase 3 — Hot-reload manifest watcher

**Hedef:** apply-bekle döngüsünü ortadan kaldır.

- `~/.cave/manifests/` (veya `--watch <dir>`) fs watcher.
- Değişiklik diff → reconcile (idempotent).
- Debounce + hata toleransı (bozuk manifest dev loop'u kırmaz).
- `cavectl dev status` watcher durumunu gösterir.

**Bağımlılık:** Phase 1 (reconcile). **Çıktı:** kaydet→otomatik-reconcile dev deneyimi.

## Phase 4 — Devcontainer template + install scripts

**Hedef:** dağıtım ve editör entegrasyonu.

- Install paths: Homebrew formula, apt/dnf paketleri, winget manifest, `curl | sh` install script — hepsi aynı tek binary'i kurar.
- Devcontainer template: VS Code / Cursor / Zed için reproducible dev environment (`cave dev up` ile bağlanır).
- Persistent state layout `~/.cave/` dokümante edilir.

**Bağımlılık:** Phase 2 (cavectl dev up). **Çıktı:** tek komutla kurulabilir, editöre bağlanabilir dev runtime.

## Phase 5 — GPU passthrough + AI auto-config

**Hedef:** local-first AI ayağı (ADR-RUNTIME-OPENJARVIS-ADOPTION-001).

- GPU detection: NVIDIA (CUDA) / AMD (ROCm) / Apple Silicon (Metal).
- Passthrough → `cave-local-llm` (Ollama/vLLM) + `cave-mlx`.
- Donanıma göre backend auto-config; algılanamayan donanımda CPU fallback + net uyarı.
- OpenJarvis backend orchestration + evaluation tools (energy/latency/cost/accuracy) bağlanır.

**Bağımlılık:** Phase 1 (component lifecycle) + ADR-RUNTIME-OPENJARVIS-ADOPTION-001 hedef crate kararı. **Çıktı:** local-first AI çalışan dev runtime.

## Platform desteği (tüm fazlar boyunca)

| Platform | Konum | Not |
|---|---|---|
| Linux native | Birinci sınıf | Gerçek cgroups/namespaces/CRI; full profile tam parite iddiası burada. |
| macOS | Fallback | Layer 1/2 native olana kadar lightweight VM / containerd-muadili datapath; "not production-equivalent". |
| Windows | Fallback | macOS ile aynı fallback modeli. |

## Önerilen ray sırası (next dispatch)

1. **Phase 1** — profile system (temel; her şey buna bağlı). *İlk dispatch.*
2. **Phase 2** — `cavectl dev` suite (developer'ın eline geçen ilk somut UX).
3. **Phase 3** — hot-reload watcher (inner-loop tamamlanır).
4. **Phase 5** — GPU + AI auto-config (OpenJarvis ayağı; Phase 1 sonrası paralel gidebilir).
5. **Phase 4** — install scripts + devcontainer (dağıtım; dev UX olgunlaştıktan sonra).

> Not: Phase 4 ve Phase 5 sıralaması esnek — Phase 5 AI vizyonu için öncelikliyse Phase 4'ten önce gidebilir. Phase 1 her durumda ilk.
