# ADR-RUNTIME-PARITY-REGEN-001 — Hourly regen of `docs/parity/parity-index.json`

**Date:** 2026-05-19
**Status:** Accepted
**Author:** Burak / Cave Runtime ops

## Context

`docs/parity/parity-index.json` is the source-of-truth that the live
`/admin/compliance` dashboard, the cave-upstream-watchd dispatcher's
`parity_ratio` field, the cavectl audit commands, and every Charter v2
audit pass read. It is regenerated by `scripts/build-parity-index.py`,
which walks every crate's manifest (`parity.manifest.toml`) plus the
audit doc `docs/parity/full-audit-2026-05-01.md` and stamps a fresh
JSON snapshot.

Until 2026-05-19 the JSON was regenerated **manually** — every audit
closer ran the script as part of their close-out commit. That worked
when audit closers were the only consumers, but the live dashboard,
the dispatcher, and the cavectl audit commands all read the JSON
between manual regens, which means they see stale ratios for hours
or days at a time. The dashboard surfaced this with rows where the
on-disk manifest had moved well past the JSON snapshot.

## Decision

A `launchd` user-agent regenerates the JSON **hourly**, in place,
into the canonical workspace at `~/Code/cave-runtime-main`.

- Plist source-of-truth: `scripts/com.cave.parity-index-regen.plist.template`
- Installed copy: `~/Library/LaunchAgents/com.cave.parity-index-regen.plist`
- `StartInterval = 3600` (hourly)
- `RunAtLoad = true` (first tick fires on bootstrap)
- `Disabled = false` (active out of the box)
- Logs: `~/Library/Application Support/cave-runtime/parity-index-regen.{log,err}`

## Why it is safe to run unsupervised

- The script only **reads** crate manifests + the audit markdown and
  **writes** the single JSON output. No git mutations, no network
  calls, no shell-out to anything that could touch a remote.
- The script is idempotent: re-running it on an unchanged workspace
  produces a byte-identical file (modulo timestamp fields in the
  generated JSON header).
- On error it exits non-zero and emits to `parity-index-regen.err`;
  the previous JSON is untouched.
- Footprint is tiny: <1s wall, <50MB RSS on a 112-crate workspace.

## Why hourly, not 5-minute / daily

- 5-minute matches the `cave-upstream-watchd` poller cadence — too
  aggressive for a derived artefact that only moves when manifests
  change (audit close-outs, usually once or twice a week).
- Daily lets the dashboard go a full work-day out-of-date during a
  busy close-out push.
- Hourly is the smallest cadence that keeps the dashboard within a
  human-perceptible window of "live" without churning disk for no
  reason.

## Linux equivalent

A `systemd` timer + service should be added under
`deploy/systemd/cave-parity-index-regen.{service,timer}` once the
runtime moves out of macOS-only ops. The script itself is portable
Python 3 — no macOS-specific calls.

## Verification

After bootstrap:

```sh
launchctl list | grep parity-index-regen
tail -5 ~/Library/Application\ Support/cave-runtime/parity-index-regen.log
python3 -c 'import json,sys; \
  d = json.load(open("docs/parity/parity-index.json")); \
  print("crates:", len(d["crates"]), "generated_at:", d["generated_at"])'
```

The `generated_at` field must be within the last hour after the
first tick.
