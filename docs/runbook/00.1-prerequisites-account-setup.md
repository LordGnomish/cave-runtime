# CAVE Platform Runbook §00.1 — Prerequisites and Account Setup

**CAVE** (Cloud-Agnostic Virtualized Environment) is an Internal Developer Platform that abstracts away cloud complexity through a unified control plane, enabling platform teams and product engineers to deploy, manage, and observe services across hybrid infrastructure. This section details the prerequisites needed to bootstrap CAVE from a blank slate, the rationale behind our architectural decisions, and step-by-step guidance for setting up accounts and credentials.

---

## 0.1.1 Overview: What Prerequisites Are Needed and Why

CAVE's prerequisites are structured around three core principles:

1. **Cloud Agnosticism**: Rather than lock users to a single cloud provider, CAVE must support both on-premises Hetzner bare-metal and managed Azure services. This requires a separate account (with its own identity provider, secrets management, and billing) for each cloud.

2. **Secret Zero Bootstrap**: Every production system faces the chicken-and-egg problem: how do you encrypt secrets if you have no initial credentials? Our answer (ADR-079) is to temporarily use unencrypted local credentials for Day 0 bootstrap only, then immediately rotate them into vault. This approach is more pragmatic than trying to initialize HashiCorp Vault from a sealed state or managing AWS SSM Parameter Store before infrastructure exists.

3. **Zero-Trust Identity**: Rather than rely on a single IdP, CAVE uses an identity split: **Okta Workforce Identity** is the master identity provider (owns human users, service accounts, SCIM sync), while **Entra ID** is used only for Azure resource RBAC. This separation (ADR-064) prevents any single identity breach from compromising both humans and infrastructure.

Because of these principles, CAVE requires accounts and tokens across six independent systems. The following sections explain each one, why we chose it, and what to do if things go wrong.

---

## 0.1.2 ADR Rationale: ADR-079 — Secret Zero Bootstrap

**Context**: Every production platform must address the bootstrap paradox. When CAVE's control plane first boots, it needs database credentials, encryption keys, and API tokens to configure itself. But those secrets must themselves be encrypted. Traditional approaches (e.g., "initialize Vault with Terraform" or "use AWS KMS to encrypt Terraform state") require either Vault or cloud infrastructure to already be running—a chicken-and-egg problem.

**Problem**:
- If we initialize Vault from a sealed state, we need the unseal keys somewhere—but where?
- If we store secrets in Terraform state, we must encrypt that state somehow—but we have no KMS yet.
- If we use SSH keys to bootstrap, we still need to store and rotate them safely.
- Manual provisioning is operationally fragile and doesn't scale to multiple clouds.

**Decision**:
CAVE uses **local `.env` files** to store initial credentials for Day 0 bootstrap *only*. These files contain plaintext secrets (database passwords, cloud API tokens, initial Vault tokens) needed to bring up infrastructure and secrets management. The `.env` file is **never committed to Git**, is **deleted immediately after bootstrap completes**, and credentials are **rotated within hours** into OpenBao (for on-prem) or Azure Key Vault (for managed). This approach trades short-term visibility risk for operational simplicity and resilience.

**Alternatives Evaluated**:

| Alternative | Rationale for Rejection |
|---|---|
| **HashiCorp Vault init** | Vault must be initialized *after* it is deployed, but we need secrets *to* deploy it. Chicken-and-egg. We could hardcode an initial root token, but that's worse than `.env` because it's harder to rotate and audit. |
| **AWS SSM Parameter Store** | AWS infrastructure (EC2, RDS, network) doesn't exist yet, so we cannot authenticate to AWS before AWS exists. Only works for cloud-first deployments, not multi-cloud. |
| **Manual provisioning** | Requires humans to SSH into machines and paste secrets, which is unmotivated, not auditable, and doesn't scale across multiple clusters or clouds. Velocity-killer. |
| **Sealed Vault + Human-held Unseal Keys** | Requires keeping unseal key shards somewhere (e.g., paper in safes). Operationally complex and doesn't solve the bootstrap problem—you still need initial credentials. |
| **Terraform Cloud/Remote State** | Pushes the problem upstream. Where do Terraform Cloud credentials live? This just moves the secret to a different system. |

**Consequences**:
- **Pro**: CAVE bootstraps in <10 minutes from zero. Operator needs only a laptop with `cave-ctl` and the `.env` file.
- **Pro**: Multi-cloud deployments use the same pattern (one `.env` per cloud, populated once, destroyed once).
- **Pro**: Credential rotation is automated and auditable (stored in vault logs).
- **Con**: For ~1 hour, credentials exist in plaintext on disk. Mitigated by: (1) `.env` is on operator's laptop, not in CI/CD; (2) bootstrap automation is fast; (3) secrets are rotated into vault automatically.
- **Con**: If bootstrap fails midway (e.g., network outage during `cave-ctl create`), operator must manually verify which credentials were rotated and which are still in `.env`. Addressed by idempotent `cave-ctl create` and manual reconciliation playbook (§00.2).

## 0.1.3 All-Profile Prerequisites

These accounts and tokens are required *regardless of which cloud* CAVE is deployed to:

| Account / Tool | Purpose | Why This Choice |
|---|---|---|
| **GitHub Organization** | Source control for platform code, Backstage entity catalog, GitHub Actions CI/CD, OAuth provider for internal services | GitHub is the de facto standard for engineering teams. Its Actions runner is tightly integrated into our GitOps workflow (ArgoCD pulls manifests from git). GitHub's OIDC support (for keyless auth to clouds) means `cave-ctl` and ArgoCD don't need to store long-lived tokens. GitHub's GraphQL API is mature for Backstage catalog sync. |
| **Cloudflare Account + API Token** | Global DNS provider, wildcard TLS certificate management (cert-manager DNS-01 ACME), DDoS protection, edge caching, edge workers for observability | Cloudflare's API-driven DNS is faster to provision than Route53 or Azure DNS (no multi-step auth delays). Cloudflare's SSL/TLS integration means our wildcard certs for `*.cave.caveplatform.dev` renew automatically without touching Kong. Cloudflare's DDoS protection is free and good. Its edge worker integration allows us to inject observability headers (e.g., request ID, trace context) at the edge without modifying application code. |
| **Domain** | Public DNS apex for CAVE, e.g., `caveplatform.dev` | Owned by the organization, not tied to any cloud provider, portable. DNS is migrated with a CNAME cutover; no re-provisioning needed if we switch clouds or add regions. |

## 0.1.4 Hetzner-Specific Prerequisites

Hetzner is CAVE's preferred on-premises and cost-optimized cloud option. It is chosen for:
- **Bare-metal philosophy**: Hetzner uses real hardware, not virtualized slices, so performance is predictable.
- **Cost**: 70% cheaper than AWS/Azure for compute + storage combined.
- **Transparency**: No hidden egress charges, no complexity tax.

**Hetzner Prerequisites**:

| Credential | Purpose | How to Obtain |
|---|---|---|
| **Hetzner Cloud API Token** | Programmatic access to provision VMs, load balancers, volumes, firewalls | Log in to Hetzner Cloud console (cloud.hetzner.com) → Projects → API Tokens → Generate Token. Scope to single project. Store in `.env` only. |
| **Hetzner Firewall** | Network policy enforcement (ingress rules for Kong, egress allowlist for registry pulls) | Created by OpenTofu templates; no manual setup. Firewall rules are versioned in git. |

**Why Hetzner (ADR-001)**:
Our platform is optimized for teams that value **operational transparency** and **cost efficiency** over managed service convenience. Hetzner gives us direct control over Linux kernel parameters, scheduling policies, and network QoS—necessary for deterministic latency testing and cost predictability. Azure managed services (AKS, Cosmos DB) abstract these details away but make performance tuning opaque and bills volatile. Hetzner is not suitable for regulatory scenarios (HIPAA, FedRAMP); those teams should use the Azure profile instead.

## 0.1.5 Azure-Specific Prerequisites

Azure is CAVE's managed cloud option, chosen for regulated environments, multi-region redundancy, and tight integration with Microsoft identity ecosystems. Setting up CAVE on Azure requires five additional prerequisites beyond the all-profile list:

| Account / Tool | Purpose | Why This Choice | Secret Management |
|---|---|---|---|
| **Azure Subscription** | Billing and container for resources (Resource Groups, AKS, Cosmos DB, Event Hubs) | Standard entry point for any Azure deployment. Usually managed by finance team. Share subscription ID with CAVE team. | Subscription ID is non-secret; billing is tracked by Entra ID tenant. |
| **Service Principal (Hetzner or Hybrid)** | When CAVE runs *on* Azure but needs to manage Hetzner infrastructure (e.g., disaster recovery nodes), a Hetzner-facing SP is created in Entra ID but credentials (API key) point to Hetzner. | Separates Azure resource auth (Entra ID) from Hetzner provisioning (token). Allows Azure-native RBAC to control who can destroy/rebuild Hetzner clusters. | Store Hetzner token in Azure Key Vault, retrieved by ArgoCD via managed identity. |
| **Okta Workforce Identity + API Token** | Master identity provider for all humans and service accounts. SCIM sync from Okta → Entra ID, Okta → GitHub. OIDC endpoint for `cave-ctl` keyless auth. | Okta is single source of truth for identity. Unlike Entra ID, Okta is cloud-agnostic; if CAVE is deployed to AWS or GCP later, we don't re-provision identity. Okta's OIDC is OpenID Foundation certified (better spec compliance than Entra ID). SCIM bidirectional sync is faster and more reliable than Graph API loops. | API token stored in .env, rotated to Azure Key Vault after bootstrap. Service account credentials stored in Okta vaults (not Azure). |
| **Entra ID App Registration** | RBAC for Azure resources only. Does NOT own humans; owned by Okta OIDC. | Entra ID's Azure RBAC is tightly integrated into Azure APIs. Using Entra ID for humans would create identity split later (Okta still owns Hetzner auth). By using Entra ID for Azure *resources only* and Okta for Azure *humans*, we avoid confusion. This is ADR-064. | Managed Identity (no shared secret). App's federated credentials trust Okta OIDC issuer; Azure validates tokens signed by Okta. |
| **Confluent Cloud Account + API Key** | Managed Kafka for event streaming (audit logs, metrics, async jobs). Confluent Schema Registry for schema versioning. | Kafka is not a commodity; it requires tuning, monitoring, and cluster rebalancing. Confluent Cloud removes this burden. Confluent Schema Registry is tightly integrated (one platform, one API, one interface). Alternatives (AWS MSK, self-managed on AKS) either lock you to AWS or require >2 FTE to keep running. | API key stored in .env, rotated to Azure Key Vault. Key rotated every 90 days (Confluent policy). |
| **Databricks Account** | Managed data platform, MLflow for model registry, managed notebooks, Kubernetes compute on AKS. | Databricks simplifies multi-cloud data pipelines (runs on Azure, AWS, GCP with one API). MLflow is the industry standard for ML ops. Databricks Unity Catalog is the only managed solution for data governance at scale. | OAuth via Okta; Databricks redirects to Okta for login. No stored credentials needed. |
| **GitHub Enterprise (or GitHub Team)** | Tenant-specific source control for tenant code. Public GitHub repos are fine for libraries and open-source; tenant-specific repos use GitHub Enterprise. | GitHub Enterprise allows customized auth (SAML via Okta), audit logging, and compliance features. Keeps tenant code separate from shared platform code. | Auth via Okta SAML. No separate credentials. |

**Identity Split Rationale (ADR-064)**:
Why do we use Okta for humans and Entra ID for Azure resources? Because:
1. If an attacker compromises Entra ID, they gain access to Azure resources only (IaC, databases, networks). They don't gain access to Okta, GitHub, Hetzner, or Databricks.
2. If Okta is compromised, attackers gain access to humans and CI/CD (GitHub Actions), but they cannot directly access Azure resources because Azure resources don't trust Okta directly—they trust the Entra ID app registration, which is a separate identity.
3. Each system has a clear separation of concern: Okta owns *human identity*, Entra ID owns *Azure resource access*.

## 0.1.6 DNS Configuration

CAVE's DNS is managed entirely by OpenTofu (Terraform). DNS records are *created automatically* when infrastructure is provisioned; operators do not manually manage DNS.

**Record Structure**:

| Record | Type | Target | Purpose |
|---|---|---|---|
| `caveplatform.dev` | A | Platform LB IP (Hetzner) or ALB IP (Azure) | Public apex; points to ingress load balancer. |
| `*.cave.caveplatform.dev` | CNAME | `caveplatform.dev` | Wildcard for tenant-specific APIs and dashboards (e.g., `tenant-a.cave.caveplatform.dev` → same LB). |
| `*.api.caveplatform.dev` | CNAME | `caveplatform.dev` | Wildcard for API gateways (Kong). Each service gets its own subdomain (e.g., `order-service.api.caveplatform.dev`). |

**Why This Structure**:
- **Single LB, Many Hosts**: All traffic (tenant dashboards, service APIs, platform APIs) goes through a single load balancer (Kong on Hetzner, Azure Application Gateway on Azure). DNS routing via hostname (not via DNS round-robin) is more resilient.
- **Wildcard TLS**: Cloudflare's wildcard cert covers `*.cave.caveplatform.dev` and `*.api.caveplatform.dev`, so no re-provisioning is needed when you add a new tenant or service. Cert renewal happens automatically via DNS-01 challenge (no human touch).
- **Kong Routing**: Kong reads the `Host` header and routes to the correct backend. A request to `tenant-a.cave.caveplatform.dev` hits Kong, Kong looks up tenant `a` in its config (stored in PostgreSQL), and routes to the tenant's namespace. This is ADR-062 (Kong as unified ingress).

**DNS Lifecycle**:
1. Operator runs `cave-ctl create --cloud hetzner`.
2. OpenTofu provisions LB, records IP in a local file.
3. OpenTofu calls Cloudflare API to create/update A record for `caveplatform.dev`.
4. CNAME records are created for wildcard subdomains.
5. Within 2 minutes, DNS propagates globally (Cloudflare's edge).
6. cert-manager runs DNS-01 ACME challenge against Cloudflare API, which creates a temporary TXT record to prove domain ownership.
7. TLS cert is issued and stored in Kubernetes secret `cert-manager-tls`.
8. Kong uses the cert to terminate TLS on all incoming HTTPS connections.

No operator needs to touch the Cloudflare dashboard.

## 0.1.7 Secret Zero Bootstrap Protocol

This section walks through exactly what happens when an operator runs `cave-ctl create` and credentials are bootstrapped.

**Step 1: Operator Prepares `.env` File**

Operator creates a file `/home/user/cave-bootstrap/.env` with the following content:

```bash
# Day 0 Bootstrap Secrets (NEVER COMMIT TO GIT)
# Deleted automatically after bootstrap completes

# Cloud provider credentials
HETZNER_API_TOKEN=<long hex string>
CLOUDFLARE_API_TOKEN=<long hex string>

# Database seeding
POSTGRES_ADMIN_PASSWORD=<32-char random>
POSTGRES_REPLICATION_PASSWORD=<32-char random>

# Vault initialization
VAULT_UNSEAL_KEY=<base64>
VAULT_ROOT_TOKEN=<base64>

# Identity provider
OKTA_API_TOKEN=<long hex string>
GITHUB_ORG_TOKEN=<long hex string>

# Crypto
ETCD_ENCRYPTION_KEY=<base64-encoded 32-byte key>
```

This file is **never committed to git**, **never stored in CI/CD**, **never backed up**. It lives only on the operator's laptop for ~30 minutes.

**Step 2: `cave-ctl create` Reads `.env` and Boots Infrastructure**

Operator runs:
```bash
cave-ctl create \
  --cloud hetzner \
  --region fsn1 \
  --env /home/user/cave-bootstrap/.env
```

`cave-ctl` (a compiled Go binary) reads the `.env` file into memory and:
1. Calls Hetzner API to provision VMs, LB, volumes, firewall.
2. Calls Cloudflare API to create DNS records.
3. Generates initial kubeconfig and stores it in `/home/user/.kube/config`.
4. Uses SSH to push the kubeconfig to the provisioned VMs.
5. Polls the API servers until they are healthy (usually 2–3 minutes).

**Step 3: Kubernetes Boots, etcd Encrypts Secrets**

Once API servers are healthy, `cave-ctl` runs:
```bash
kubectl apply -f /opt/cave/manifests/vault-bootstrap.yaml
```

This manifest creates:
- A single-node OpenBao (Vault fork) pod with storage backend = local etcd.
- Database operator that creates a PostgreSQL admin user + password.
- cert-manager that provisions the TLS certificate from Cloudflare.

All secrets written to etcd are encrypted using the `ETCD_ENCRYPTION_KEY` from `.env`.

**Step 4: Credentials Are Rotated Into Vault**

Once Vault is healthy, `cave-ctl` runs:
```bash
curl -X POST http://vault.cave.svc.cluster.local:8200/v1/secret/data/bootstrap \
  -H "X-Vault-Token: $(cat /home/user/.kube/vault-root-token)" \
  -d @- <<EOF
{
  "data": {
    "hetzner_api_token": "$HETZNER_API_TOKEN",
    "postgres_password": "$POSTGRES_ADMIN_PASSWORD",
    "okta_api_token": "$OKTA_API_TOKEN",
    ...
  }
}
EOF
```

Once this succeeds (Vault now owns the secrets), `cave-ctl` updates ArgoCD's secret store:
```bash
kubectl patch secret argocd-repo-creds-default \
  -p '{"stringData": {"password": "vault:secret/data/bootstrap#hetzner_api_token"}}'
```

This tells ArgoCD to fetch Hetzner credentials from Vault, not from `.env`.

**Step 5: `.env` Is Destroyed**

Finally, `cave-ctl` runs:
```bash
shred -vfz -n 10 /home/user/cave-bootstrap/.env
```

The file is overwritten 10 times with random data and deleted. It no longer exists on disk.

**Step 6: Validate Bootstrap Success**

`cave-ctl` validates:
- Vault is unsealed and healthy.
- PostgreSQL is running and initialized.
- ArgoCD is connected to Vault and can pull secrets.
- Cert-manager has provisioned a valid TLS cert.
- Kong is routing traffic correctly.

If any validation fails, `cave-ctl` prints a detailed error message and does **not** delete `.env`. Operator must investigate and can re-run bootstrap once the issue is fixed.

**Idempotency**: `cave-ctl create` is idempotent. If run twice with the same arguments, it will:
1. Check if infrastructure already exists (by Hetzner API tag).
2. If it exists, skip VM provisioning.
3. Re-apply Kubernetes manifests (safe due to kustomize + strategic merge).
4. Re-rotate credentials into Vault (overwrites previous secrets).

## 0.1.8 Offline Break-glass Kit

Even with perfect operations, CAVE's control plane can fail catastrophically (etcd corruption, Vault sealed permanently, network partition lasting hours). The offline break-glass kit is a set of physical and digital artifacts that allow the platform team to recover CAVE to a healthy state without external dependencies.

**Contents of the Break-glass Kit**:

| Artifact | Format | Storage | Rotation | Access |
|---|---|---|---|---|
| Vault unseal keys (5 keys) | Paper (printed QR code + hex) | Shamir 3-of-5 split: 2 keys in physical safe at HQ, 1 key at co-location, 2 keys with remote team leads | Annually (during resurrection drill) | Requires opening physical safe + two remote signers |
| Cloudflare root token | Paper (printed 40-char string) | Physical safe | Annually | One designated sec ops person |
| Hetzner API token | Paper (printed 40-char string) | Physical safe | Annually | One designated sec ops person |
| Okta API token | Paper (printed 40-char string) | Physical safe | Annually | One designated sec ops person |
| etcd encryption key | Paper (base64-encoded 32 bytes) | Physical safe | Annually | One designated sec ops person |
| Static snapshot of Vault schema | USB drive (encrypted, key in safe) | Physical safe | Annually | Same access as Vault keys |

**Key Shamir Scheme**:
Vault unseal keys use Shamir's secret sharing (3-of-5 threshold):
- Generate 5 unseal key shards using `vault operator rekey`.
- Shard 1: Safe at HQ (SFO office).
- Shard 2: Safe at HQ (SFO office).
- Shard 3: Co-location (AWS us-east-1).
- Shard 4: Held by team lead A (encrypted, in password manager).
- Shard 5: Held by team lead B (encrypted, in password manager).

To unseal Vault: fetch 3 shards from 3 different locations and run `vault operator unseal`.

**RTO and Remote HSM Access**:
The goal is to restore CAVE to a healthy, production state in <4 hours (measured from incident declaration to ArgoCD reconciling all manifests). To achieve this:
- One key holder must have remote access to an HSM (YubiHSM or Azure Dedicated HSM) that stores an encrypted copy of the Vault unseal key.
- This HSM can be unlocked remotely (via VPN + TOTP) without physical travel.
- Reduces RTO from "fly person to safe" (8–12 hours) to "unlock HSM" (30 minutes).

**Annual Resurrection Drill**:
Every 12 months, the platform team conducts a resurrection drill:
1. Tear down CAVE cluster (delete all infrastructure).
2. Attempt to rebuild from the break-glass kit alone (no `.env` files, no recent backups).
3. Measure time to healthy state.
4. Update key holders, validate all keys are still usable.
5. Document any gaps and update the kit.

This drill is mandatory for compliance (SOC 2 CA) and catches operational decay (e.g., a key holder leaves the company, but we don't notice until disaster strikes).

**Connection to §36 Resurrection Protocol**:
The break-glass kit is the physical foundation of the resurrection protocol (§36), which defines the detailed playbook for restoring CAVE after infrastructure loss or data loss. The kit must be maintained and validated continuously.

## 0.1.9 MCP Server Access (Azure Profiles)

MCP (Model Context Protocol) servers extend CAVE's control plane to integrate with external systems (cloud APIs, data platforms, identity providers). This section details how MCPs authenticate to these systems.

**MCP Topology**:
Each MCP server is a containerized binary running inside Kubernetes, authenticated via a credential type:

| MCP Server | Credential Type | Provider | Why This Method |
|---|---|---|---|
| **Azure Terraform Provider MCP** | Managed Identity (Azure) | Entra ID App Registration | No shared secrets stored in K8s. Azure's Managed Identity API is called by kubelet; credentials are injected at the pod level. Scoped to specific Azure RBAC roles. |
| **GitHub MCP** | GitHub App (OAuth2) | GitHub | App-level auth means MCP credentials are tied to a specific GitHub App, not a personal token. Revocation or key rotation is scoped to the app. GitHub App has fine-grained permissions (read-only for public repos, read-write for platform repos). |
| **Confluent MCP** | API Key (static) | Confluent Cloud | Confluent doesn't support managed identities. API key is stored as a Kubernetes Secret, encrypted at rest in etcd (using `ETCD_ENCRYPTION_KEY`). Key is rotated every 90 days by a CronJob that calls Confluent API. |
| **Databricks MCP** | OAuth via Okta | Databricks + Okta | Databricks OAuth redirects to Okta OIDC. MCP doesn't store credentials; it uses Okta token (fetched fresh every hour). Token is short-lived (1 hour), reducing blast radius of token theft. |

**MCP Lifecycle**:
1. Operator creates MCP Server manifest in git: `platform/mcp-servers/github-mcp.yaml`.
2. Manifest specifies credential source: `credentialSource: managedIdentity` or `credentialSource: secret`.
3. Platform Operator (a CAVE controller) reads the manifest and:
   - If `managedIdentity`: creates a Kubernetes ServiceAccount with Azure Workload Identity annotations.
   - If `secret`: creates a Kubernetes Secret with encrypted credential material.
4. MCP Pod is scheduled on a node, kubelet injects the credential.
5. MCP calls external API using the credential.

## 0.1.10 Account Setup Walkthrough

This section provides step-by-step guidance for setting up CAVE on two popular clouds: Hetzner and Azure.

### 0.1.10.1 Hetzner Deployment Walkthrough

**Prerequisites**:
- GitHub organization created (organization name: `mycompany-platform`, at least one member with admin role).
- Cloudflare account created (one zone: `mycompany.dev`, DNS delegation completed).
- Domain `mycompany.dev` already registered and pointed to Cloudflare nameservers.
- Hetzner Cloud account created (billing method added, at least one project created).

**Step 1: Gather Credentials**

Operator collects all credentials:

```bash
# GitHub: Generate a new OAuth App
# Settings → Developer settings → OAuth Apps → New OAuth App
# Authorization callback URL: https://cave.mycompany.dev/callback
# Copy: Client ID and Client Secret

# Cloudflare: Generate API Token
# Settings → API Tokens → Create Token → Edit zone DNS
# Copy: API Token (40-char string)

# Hetzner: Generate API Token
# Console → Projects → myproject → Security → API Tokens → Generate token
# Copy: API Token (long hex string)

# Okta: Generate API token
# Admin → API → Tokens → Create token
# Copy: API Token (long hex string)

# Generate initial credentials
POSTGRES_ADMIN_PASSWORD=$(openssl rand -base64 32)
POSTGRES_REPLICATION_PASSWORD=$(openssl rand -base64 32)
VAULT_UNSEAL_KEY=$(openssl rand -base64 32)
VAULT_ROOT_TOKEN=$(uuidgen)
ETCD_ENCRYPTION_KEY=$(openssl rand -base64 32)

mkdir -p ~/cave-bootstrap
cat > ~/cave-bootstrap/.env <<EOF
HETZNER_API_TOKEN=<paste from Hetzner console>
CLOUDFLARE_API_TOKEN=<paste from Cloudflare console>
GITHUB_ORG_TOKEN=<paste from GitHub console>
OKTA_API_TOKEN=<paste from Okta console>
POSTGRES_ADMIN_PASSWORD=$POSTGRES_ADMIN_PASSWORD
POSTGRES_REPLICATION_PASSWORD=$POSTGRES_REPLICATION_PASSWORD
VAULT_UNSEAL_KEY=$VAULT_UNSEAL_KEY
VAULT_ROOT_TOKEN=$VAULT_ROOT_TOKEN
ETCD_ENCRYPTION_KEY=$ETCD_ENCRYPTION_KEY
DOMAIN=mycompany.dev
REGION=fsn1
ORGANIZATION=mycompany-platform
EOF

chmod 600 ~/cave-bootstrap/.env
```

**Step 2: Bootstrap Infrastructure**

```bash
cave-ctl create \
  --cloud hetzner \
  --env ~/cave-bootstrap/.env \
  --dry-run  # First, validate without actually provisioning

# Review the output, then run without --dry-run
cave-ctl create \
  --cloud hetzner \
  --env ~/cave-bootstrap/.env
```

`cave-ctl` will print:
```
[+] Provisioning infrastructure (this may take 2-3 minutes)...
[+] Waiting for API servers to be healthy...
[+] Initializing Vault...
[+] Rotating credentials into Vault...
[+] Validating ArgoCD connectivity...
[+] Creating initial TLS certificate...
[+] Destroying .env file...
[✓] CAVE cluster is ready!

Kubeconfig saved to: /home/user/.kube/config
Web UI available at: https://cave.mycompany.dev
```

**Step 3: Validate Deployment**

```bash
# Check that Kubernetes API is healthy
kubectl cluster-info

# Check that all CAVE system pods are running
kubectl get pods -n cave-system
# Should show: vault, postgres, argocd, cert-manager, kong, etc.

# Check that DNS is working
nslookup cave.mycompany.dev
# Should return the LB IP

# Check that TLS cert is valid
curl -v https://cave.mycompany.dev/health
# Should return 200 OK with valid cert
```

**Step 4: Create First Tenant**

```bash
cave-ctl tenant create \
  --name "tenant-a" \
  --github-team "mycompany-platform/team-a"

# This creates:
# - Kubernetes namespace: tenant-a
# - ArgoCD AppProject: tenant-a
# - Kong service: tenant-a.cave.mycompany.dev
# - Database: tenant-a (in PostgreSQL)
# - RBAC bindings (Okta groups → Kubernetes RBAC)
```

**Step 5: Add Initial GitOps Configuration**

Operator creates a git repo for the tenant:
```bash
git clone https://github.com/mycompany-platform/platform-bootstrap
cd platform-bootstrap
mkdir -p tenants/tenant-a/overlays/prod
cat > tenants/tenant-a/overlays/prod/kustomization.yaml <<EOF
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
bases:
  - ../../base
patchesStrategicMerge:
  - replica-patch.yaml
EOF
```

Commit to git. ArgoCD auto-syncs within 30 seconds.

### 0.1.10.2 Azure Deployment Walkthrough

**Prerequisites**:
- Azure Subscription created (billing linked, at least one Resource Group: `cave-rg`).
- GitHub organization created.
- Cloudflare account created.
- Okta Workforce Identity tenant created (admin user configured).
- GitHub Enterprise created (SAML auth configured to Okta).

**Step 1: Create Azure Service Principal**

```bash
# In Azure CLI:
az ad sp create-for-rbac \
  --name "cave-platform" \
  --role "Contributor" \
  --scopes /subscriptions/<subscription-id>/resourceGroups/cave-rg

# Output:
# {
#   "appId": "...",
#   "displayName": "cave-platform",
#   "password": "...",
#   "tenant": "..."
# }
```

Store the `appId`, `password`, and `tenant` in `.env`.

**Step 2: Create Entra ID App Registration**

Instead of using a shared secret, configure federated credentials to trust the Okta OIDC issuer:

```bash
# In Azure CLI:
az ad app create --display-name "cave-kubernetes"

# Capture the <app-object-id>

az ad app federated-credential create \
  --id <app-object-id> \
  --parameters '{
    "name": "okta-oidc",
    "issuer": "https://mycompany.okta.com",
    "subject": "system:serviceaccount:cave-system:argocd",
    "audiences": ["api://cave-platform"]
  }'

# This tells Entra ID: "Trust any token from Okta OIDC that claims to be the argocd service account"
```

**Step 3: Create Kubernetes Cluster (AKS)**

```bash

  --resource-group cave-rg \
  --name cave-cluster \
  --node-count 3 \
  --vm-set-type VirtualMachineScaleSets \
  --load-balancer-sku standard \
  --enable-managed-identity \
  --network-plugin azure \
  --vnet-subnet-id /subscriptions/.../subnets/aks-subnet \
  --docker-bridge-address 172.17.0.1/16 \
  --service-principal <appId> \
  --client-secret <password>

# Get kubeconfig
az aks get-credentials \
  --resource-group cave-rg \
  --name cave-cluster \
  --admin
```

**Step 4: Gather and Prepare Credentials**

```bash
mkdir -p ~/cave-bootstrap
cat > ~/cave-bootstrap/.env <<EOF
AZURE_SUBSCRIPTION_ID=<subscription-id>
AZURE_TENANT_ID=<tenant-id>
AZURE_CLIENT_ID=<app-id>
AZURE_CLIENT_SECRET=<password>

```

**Step 5: Bootstrap Using cave-ctl**

```bash

  --cloud azure \
  --env ~/cave-bootstrap/.env

# Output is same as Hetzner deployment
```

## 0.1.11 Tool Comparison: DNS Providers

When CAVE was designed, five DNS providers were evaluated. This section details the scoring matrix and rationale for choosing Cloudflare.

| Criterion | Cloudflare | Route53 | Azure DNS | Google Cloud DNS | TransIP |
|---|---|---|---|---|---|
| **API Latency** (average time to update record) | 50ms | 200ms | 150ms | 100ms | 300ms |
| **DNS Query Latency** (user queries) | 20ms (Anycast edge) | 50ms | 40ms | 30ms | 80ms |
| **Cert-manager Integration** | Native DNS-01 provider | Native; requires IAM role | Native; requires Service Principal | Native; requires GKE auth | No native support |
| **Cost** (per month, 1M queries) | $20 (free tier for DNS) | $50 | $50 | $30 | $40 |
| **DDoS Protection** | Free (L4-L7) | Extra ($3k/yr) | Extra ($5k/yr) | Extra ($2k/yr) | Extra ($1.5k/yr) |
| **Egress Bandwidth** (to external origin) | Charged | Charged | Free | Charged | Charged |
| **Wildcard CNAME Support** | Yes | Yes | Yes | Yes | Yes |
| **DNSSEC** | Yes | Yes | Yes | Yes | Yes |
| **SLA** | 99.95% | 99.99% | 99.99% | 99.99% | 99.90% |
| **Multi-cloud Portability** | Yes (cloud-agnostic) | AWS-only (can migrate to Route53) | Azure-only | GCP-only | cloud-agnostic |
| **Edge Integration** (inject headers, cache policies) | Yes (Cloudflare Workers) | No | No | No | No |

**Chosen Provider: Cloudflare**

**Rationale**:
1. **Cost efficiency**: Free DNS tier (with SSL/TLS upgrade) means CAVE teams don't incur DNS costs for small deployments. Route53 and Azure DNS charge for every million queries; at scale, this adds up.
2. **DDoS protection included**: Cloudflare's DDoS protection is free and competitive with expensive AWS Shield options. This is a significant advantage for APIs exposed to the internet.
3. **Cloud agnosticism**: Unlike Route53 or Azure DNS, Cloudflare is not tied to a cloud provider. CAVE can migrate from Hetzner → Azure → AWS without changing DNS provider or losing cached records.
4. **Edge workers for observability**: Cloudflare allows injecting custom headers (e.g., request ID, trace context) at the edge, enabling better observability without application code changes. This is unique to Cloudflare.
5. **cert-manager integration**: cert-manager's Cloudflare provider is mature and widely used (battle-tested in thousands of clusters).

**Migration Path** (if needs change):
If an organization needs Azure DNS (e.g., for compliance reasons), CAVE's DNS layer is abstracted via OpenTofu. Changing providers requires only:
1. Update `terraform/dns-provider.tf`: change provider from `cloudflare` to `azurerm`.
2. Update cert-manager's ClusterIssuer: change `cloudflare` to `azuredns`.
3. Re-apply via `cave-ctl apply`.

No application code changes needed.

## 0.1.13 Troubleshooting

| Issue | Symptoms | Root Cause | Fix |
|---|---|---|---|
| **Vault won't unseal after bootstrap** | `cave-ctl create` hangs at "Initializing Vault", then times out after 10 min. | (1) Vault pod crashed due to OOM (not enough disk for raft log). (2) Unseal key is corrupted. | (1) Check pod logs: `kubectl logs vault-0 -n cave-system`. (2) Re-run `cave-ctl create` (idempotent); if still fails, manually unseal: `vault operator unseal $(cat /home/user/.kube/vault-unseal-key)`. |
| **DNS records not updating** | A record for `cave.mycompany.dev` still points to old LB IP after `cave-ctl create`. | (1) Cloudflare API token is invalid or expired. (2) API rate limit hit (Cloudflare allows 1200 req/min for free tier). | (1) Verify token: `curl -H "Authorization: Bearer $CLOUDFLARE_API_TOKEN" https://api.cloudflare.com/client/v4/user/tokens/verify`. (2) Wait 1 hour, re-run `cave-ctl apply`. |
| **cert-manager can't get TLS cert** | Pod logs: `ACME challenge failed: DNS validation timeout`. | (1) Cloudflare account doesn't own the domain (different account). (2) DNS propagation slow due to TTL caching. | (1) Verify domain: `dig mycompany.dev NS` should show Cloudflare nameservers. (2) Lower TTL to 60 seconds before cert-manager runs, then restore after cert is issued. |
| **PostgreSQL replication lag** | `cave-ctl health` shows "Replication lag: 15s". Tenant queries slow. | (1) Replication user password changed in `.env` but not rotated into Vault. (2) Network latency between nodes. | (1) Run `cave-ctl secrets rotate --secret postgres-replication-password`. (2) Check node-to-node latency: `ping <replica-ip>`. If >10ms, consider same-zone deployment. |
| **Kong can't find tenant backend** | `curl tenant-a.cave.mycompany.dev/api/v1/... → 502 Bad Gateway`. | (1) Kong config is stale (hasn't synced from Kubernetes). (2) Tenant namespace doesn't exist or service isn't running. | (1) Restart Kong pods: `kubectl rollout restart deployment/kong -n cave-system`. (2) Check tenant namespace: `kubectl get ns tenant-a`. Check services: `kubectl get svc -n tenant-a`. |
| **.env file wasn't deleted after bootstrap** | Files still exist in `~/cave-bootstrap/` after 1 hour. | `cave-ctl create` failed before reaching the "destroy .env" step. | (1) Check `cave-ctl` logs to find failure point. (2) Manually delete: `shred -vfz -n 10 ~/cave-bootstrap/.env`. (3) Investigate root cause (was something missing from .env?). |
| **Okta SCIM sync fails** | No users appear in Kubernetes RBAC after sync. Okta logs show "HTTP 401 Unauthorized". | (1) Okta API token is invalid or revoked. (2) SCIM app in Okta points to wrong Kubernetes endpoint. | (1) Generate new Okta token, update Secret: `kubectl patch secret okta-api-token -n cave-system --type=json -p='[{"op": "replace", "path": "/data/token", "value": "..."}]'`. (2) Verify app settings: Admin → Apps → CAVE Platform → Provisioning → Base URL should be `https://cave.mycompany.dev/okta/scim`. |
| **ArgoCD can't pull from private GitHub repo** | ArgoCD status shows "failed to fetch repository". | (1) GitHub token has expired or doesn't have `repo` scope. (2) GitHub repo doesn't exist yet. | (1) Generate new GitHub token with `repo` + `read:org` scopes. (2) Create repo: `git init && git remote add origin https://github.com/mycompany/platform-config.git && git push`. |
| **etcd corruption detected** | `cave-ctl health` shows "etcd: leader unhealthy". Kubernetes API is slow or unresponsive. | (1) Disk space exhausted. (2) Power loss during etcd write (rare). | (1) Check disk: `df -h /var/lib/etcd`. If >95%, drain and remove a node. (2) If data is corrupted beyond repair, restore from backup using break-glass kit (§0.1.8). |
| **Managed Identity auth fails on Azure** | Pod logs: `error: failed to get token from Azure IMDS: deadline exceeded`. | (1) Pod doesn't have Workload Identity annotation. (2) Azure IMDS service is slow or unavailable (rare). | (1) Check pod spec: `kubectl get pod <pod-name> -o yaml | grep serviceAccountName`. Verify ServiceAccount has annotations: `kubectl describe sa <sa-name>`. (2) Wait 30s and retry. If still failing, check Azure MSI status: `az identity show --name cave-platform ...`. |
| **Confluent API key rotation fails** | CronJob logs: `403 Forbidden: Key not valid`. Tenant can't produce to Kafka. | (1) Old key was already revoked in Confluent. (2) CronJob lacks permissions to update the Secret. | (1) Manually create new key in Confluent UI, update Secret. (2) Check RBAC: `kubectl auth can-i update secret --as=system:serviceaccount:cave-system:confluent-rotator`. Grant if missing. |

| **Okta as Master IdP** | CC6.2 (Authorization) | A.9.1.1 (Access control) | 5.5 (Physical security) | 5 (Legal basis) | Okta is the source of truth for human identity. ## Summary

CAVE's prerequisites are carefully chosen to balance operational simplicity, security, cost efficiency, and multi-cloud portability. The Secret Zero Bootstrap pattern (ADR-079) avoids the chicken-and-egg problem of secrets management while maintaining a small blast radius. The identity split between Okta and Entra ID ensures that no single provider compromise can fully breach the system. The break-glass kit provides offline, human-led recovery paths for catastrophic failure scenarios.

Operators should understand that prerequisites are not just a checklist to complete; they represent intentional design decisions captured in ADRs. Before deviating from this path (e.g., using HashiCorp Vault instead of OpenBao, or Route53 instead of Cloudflare), consult the relevant ADR and the platform team.
