## Sweep-005 — `cave_kernel::circuitbreaker` adoption

### Candidate sites

- `crates/cave-gateway/src/circuit_breaker.rs` — 169 LOC.
  `CircuitBreakerRegistry` keyed by `Uuid`. State machine tracks
  consecutive failures + consecutive successes vs. configurable
  thresholds.
- `crates/cave-mesh/src/circuit.rs` — 246 LOC. Keyed by `(host,
  Option<subset>)`. State machine tracks `consecutive_errors` with
  exponential ejection backoff (`base_ejection_time`,
  `max_ejection_time`, `max_ejection_percent`) — the Envoy/Istio
  outlier-detection model.

### Why not adopted

The kernel's `cave_kernel::circuitbreaker` is a **resilience4j**-style
breaker:

- `BreakerConfig { window, failure_rate_threshold, minimum_calls,
  reset_timeout, half_open_permitted }`.
- `window: WindowKind::{Count(n), Time(d)}` — windowed failure-rate.
- Closed → Open trigger: failure **rate** within window ≥ threshold AND
  minimum_calls met.

The two local breakers use **consecutive-failure counting**, not
windowed failure-rate. A drop-in replacement would change the trigger
semantics:

| Local breaker | Kernel breaker |
|---------------|----------------|
| Open after N consecutive failures | Open after rate ≥ threshold over last N calls |
| HalfOpen → Closed: K consecutive successes | HalfOpen → Closed: K permitted trial successes |
| Per-host exponential ejection backoff (cave-mesh) | Fixed `reset_timeout` |

Switching cave-gateway means its tests (and any operator dashboards
keyed off "consecutive failures") need re-baselining. Switching
cave-mesh means losing exponential ejection backoff (an Istio-faithful
behaviour the parity audit explicitly tracks). Both are real
behavioural deltas, not just refactors.

### What a proper Sweep-005 would do

1. Add an `Outlier`-style backoff variant to
   `cave_kernel::circuitbreaker::BreakerConfig` so it can express both
   the resilience4j and Envoy models.
2. Migrate cave-gateway first (smaller surface, no Istio parity claim).
3. Migrate cave-mesh second, with parity tests guarding ejection
   behaviour.
4. Each as a separate PR; not lumped together.

Estimated honest budget: ~6–8 hours focused.

## Sweep-006 — `cave_kernel::ratelimiter` adoption

### Candidate sites

`rg 'RateLimiter|TokenBucket|LeakyBucket|rate_limit'` finds active
rate-limit logic in:

- `cave-gateway/src/plugins/rate_limiting.rs`
- `cave-mesh/src/rate_limit.rs`
- `cave-llm-gateway/src/rate_limit.rs`
- `cave-streams/src/quota.rs`
- `cave-trace/src/sampling.rs`

### Why not adopted

The kernel's `cave_kernel::ratelimiter` ships **token-bucket** and
**leaky-bucket** primitives keyed by tenant. The five sites use:

- cave-gateway: per-route token bucket with header-extractor key.
- cave-mesh: per-destination concurrency limit (semaphore-flavoured),
  not a true token bucket.
- cave-llm-gateway: per-tenant *cost*-weighted bucket (tokens-per-second
  scaled by model price tier) — kernel has no cost-weighted variant.
- cave-streams: per-topic quota with replenishment cadence — different
  shape again.
- cave-trace: probabilistic sampling, NOT a rate limiter (mis-named
  upstream).

Five sites, five different shapes. Honest adoption would require
extending the kernel primitive (cost-weighting at minimum) and then
migrating each call site individually, gated on parity tests.

### What a proper Sweep-006 would do

1. Decide whether the kernel grows a `CostWeightedTokenBucket` or
   whether cave-llm-gateway keeps its specialised local impl.
2. Migrate cave-gateway and cave-mesh first (closest semantic match).
3. Hold cave-streams and cave-llm-gateway for a follow-up that justifies
   any kernel API growth.

Estimated honest budget: ~8–12 hours focused.
